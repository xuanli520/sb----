# 小说平台改造：无工时实施计划

状态：草案 0.1。
本计划以依赖关系和验收关口推进，不使用传统人时、人数或日历工期估算。

## 执行原则

每个阶段只有在列出的证据完成并被确认后才算完成。后续可将每项工作拆分为 Codex agent 任务，但不能因为经过了某个时间点就自动推进阶段。

## 阶段 0：冻结产品基线

**目标：** 将当前功能清单转化为已批准的实施范围。

**产出：**

- 对 [需求确认稿](01-requirements-confirmation.md)、[目标架构与迁移方案](02-architecture-and-migration.md) 和 [需求决策回执](04-decision-response.md) 中 D-01 至 D-12 的已批准答复。
- 已确认的前端源码授权/fork 许可和固定版本。
- 已选择的若依发行版/版本、认证方案、部署限制和外部集成模式。
- 角色权限矩阵、作品/章节/订单状态机和指标口径。
- 明确的首期验收范围和延期功能清单。

**退出关口：** 不存在仍会改变身份、支付、内容可见性或仓库拓扑的未决问题。

## 阶段 1：建立新产品基础能力

**目标：** 用可重复的全栈基础替换演示项目基线。

**计划范围：**

- 创建已批准的 RuoYi 模块树和 Next.js 应用边界。
- 建立版本化 Schema、演示数据，以及 MySQL/Redis/本地对象存储开发拓扑。
- 定义 API 契约和前端 DTO/分页/错误适配层。
- 实现若依角色/权限骨架和已批准的同源 BFF 会话设计，包括 Redis BFF 会话、注销转发、Origin/CSRF 防护。
- 建立用户端、作者端、运营端的路由布局，移除前端模板中的抖音领域导航和服务。
- 建立环境示例、密钥政策、lint/typecheck/test/build 质量关口、健康检查和容器部署拓扑。

**退出关口：** 干净环境可启动前后端；测试账号能够认证；后端可强制执行示例权限；自动化质量检查可通过。

## 阶段 2：交付核心读者路径

**目标：** 使读者能够安全地发现和消费已发布内容。

**计划范围：**

- 已确认的注册/登录方式、资料、签到积分和账号状态处理。
- 分类/标签/作品/卷/章节的发布读取模型。
- 首页推荐、热榜、筛选和已确认的搜索/高亮行为。
- 响应式阅读器、阅读偏好、目录、书架、书签和进度同步。
- 第一批已批准互动：评分、评论、划线/段评及其审核状态。
- 兑换码核销、代币账本、整本购买、会员权益和打赏的最小闭环；不接入线上支付。

**退出关口：** 读者能够完成“登录 -> 发现 -> 阅读 -> 恢复进度 -> 允许的互动”全链路；归属与权限测试覆盖该路径。

## 阶段 3：交付作者路径

**目标：** 使获得授权的作者在不绕过审核规则的前提下创作和提交内容。

**计划范围：**

- 作者申请/资料和作者端路由/API 授权。
- 作品、卷、章节编辑；草稿、修改、定时发布、提交审核和驳回反馈。
- 上架申请规则及经确认的作品/章节状态机。
- 作者评论/打赏视图和已定义口径的作者数据看板。

**退出关口：** 作者可创建作品、管理卷和章节、提交章节、收到通过/驳回结果；且无法访问其他作者资源。

## 阶段 4：交付运营路径

**目标：** 使内容可见性、安全和系统配置均可控、可审计。

**计划范围：**

- 本地敏感词、Qwen 自动筛查、完整作品人工审核、审核审计、违规处理和账号状态操作。
- 推荐位、分类/标签、敏感词、兑换码策略和热搜词管理。
- 按已批准指标定义建设全站、渠道和留存统计流水。
- 若依菜单/权限配置、操作日志和审核/运营数据范围。

**退出关口：** 有权限的运营人员可完成并审计审核或配置动作，读者/作者端的可见性变化严格符合状态机。

## 阶段 5：实现已批准的外部与高级能力

**目标：** 只在凭据、规则和运行责任明确后，将可选/高级能力接入为可验证的生产能力。

**计划范围（获得批准时）：**

- 手机/邮箱验证、微信/QQ OAuth 和账号绑定。
- 当质量/规模目标要求时，引入专用搜索、Qwen 模型升级或高级内容审核。
- 当兑换码运营规则扩展时，增加批次导入、渠道统计和更细粒度权益策略。

**退出关口：** 每个供应商或模型流程均有自动化契约覆盖、失败/降级处理、审计记录和明确的运营责任人。

## 阶段 6：验证、部署与验收

**目标：** 在目标环境中证明已确认产品范围可用。

**计划验证：**

- 后端单元/服务/数据访问测试，覆盖状态迁移、授权、兑换码核销幂等和账本不变量。
- Next.js 适配层与若依/小说接口之间的 API 契约测试。
- 用户端、作者端、运营端的前端组件、可访问性和端到端测试。
- Qwen 审核的结构化输出、超时、限流、无效 JSON、人工复核降级和提示词版本审计测试。
- 授权绕过、对象归属、输入校验、限流、密钥、上传和审计记录的安全检查。
- 容器启动、迁移、备份恢复、健康检查、日志和部署冒烟测试。
- 按已确认浏览器/设备矩阵验证，包括阅读器的翻页模式。

**退出关口：** 所有已接受场景通过；延期范围被明确记录；部署/恢复说明可重复执行；仓库不包含密钥或生成依赖产物。

## 依赖关系图

```text
阶段 0：范围与技术决策确认
  -> 阶段 1：基础能力与契约
    -> 阶段 2：核心读者路径
    -> 阶段 3：作者工作流
    -> 阶段 4：运营工作流
      -> 阶段 5：外部/高级能力
        -> 阶段 6：验收与部署验证
```

阶段 2、阶段 3 和阶段 4 的部分工作，只能在阶段 1 固定共享身份、内容目录、API 和状态迁移契约后并行。OAuth、Qwen 生产审核和高级搜索，在未取得供应商接入和明确运行责任之前，不能标记为完成；兑换码不依赖线上支付网关。

## 文档维护规则

- 每一个范围决定及其理由，必须先记录到需求确认稿，再调整实现范围。
- API 契约、数据库迁移、权限字符串和状态机必须与实现它们的代码一同纳入版本控制。
- 不得将旧作者原型的 `ddl-auto:update` 当作数据库迁移策略。
- 不得提交密码、百炼/API 密钥、OAuth 密钥、前端构建产物、`node_modules` 或本地数据库数据。
