# 小说平台改造：需求确认稿

状态：草案 0.2，核心范围与架构已确认；外部账号供应商、部署参数和数值型运营规则在实施前配置。
本文范围：仅进行需求梳理和方案规划，不包含任何应用代码改造，也不包含传统人时或工期估算。

## 1. 已确认的目标

目标产品是一个在线小说平台，包含三个相互独立的使用体验：

1. 面向读者的用户端（前台）。
2. 面向创作者的作者端。
3. 面向内容与运营人员的运营管理后台。

已给出的技术约束如下：

- 后端采用 Spring Boot，并基于若依框架。
- 前端采用 Next.js，并以 `xuanli520/douyin_dashboard_frontend` 为基础。
- 当前阶段只产出需求、架构和实施计划文档；后续实施过程由 Codex agent 推进，不以传统工时作为计划依据。

当前功能来源为 [requirment.md](../requirment.md)。该文件是功能清单，不是完整 PRD；已确认决策会在本文中固化，尚未给出外部凭据或数值的部分作为可配置项处理。

集中回复方式见 [需求决策回执](04-decision-response.md)。

## 2. 现状与迁移边界

| 资产 | 审计结论 | 建议处理方式 |
| --- | --- | --- |
| 当前 Git 项目 | 这是 Spring Boot + Thymeleaf 的 ASCII 地球演示项目，没有小说业务、数据库、若依或 Next.js 代码。 | 以绿地重建方式改造；保留 Git 历史，但不迁移 Thymeleaf/SSE 演示功能。 |
| `novel-author.zip` | 未纳入版本控制的 JPA/MySQL 作者端原型，包含 `User`、`Novel`、`Chapter`、`Comment`、`Bookshelf` 和基础作者接口。 | 未得到明确指令前，只将其作为领域术语和流程参考，不能原样迁入若依。 |
| `xuanli520/douyin_dashboard_frontend` | Next.js 管理台工程模板，含可复用 UI、请求、状态、测试与部署基础设施；业务语义是抖音运营，不是小说平台。 | 复用工程骨架和通用组件，重建小说领域页面、导航、接口、类型和权限；导入前固定审查过的版本。 |

## 3. 角色与授权模型

建议采用“一个账号，可拥有多个角色”的统一身份模型，而不是为读者、作者和后台人员建立多套账号体系。

| 角色 | 主要职责 | 授权边界 |
| --- | --- | --- |
| 读者 | 发现和阅读小说，管理书架、阅读进度和书签，互动与消费。 | 只能操作自己的资料、资产、订单、书架、评论和互动内容。 |
| 作者 | 创建作品，管理卷和章节，提交审核，查看自身数据，处理自己作品下的互动。 | 只能操作已验证归属自己的作品和相关数据。 |
| 站长（超级管理员） | 审核完整作品、处理违规内容，管理推荐位、分类、敏感词、兑换码、报表、角色和系统配置。 | 拥有全局范围，关键动作必须记录审计日志。 |

- **D-05 已确认：** 采用邀请制作者申请。读者提交书面申请，只有站长可审核；通过后授予作者角色。驳回后可再次提交，具体冷却期作为配置项。
- **D-06 已确认：** 首期不设置独立审核员、编辑、客服、财务或分析师角色；所有运营和审核操作由站长（超级管理员）承担。

## 5. 建议的状态机

这些状态机用于消除功能清单中的歧义，必须在建模和接口设计前确认或调整。

### 5.1 作品与章节发布

```text
作品：草稿 -> 提交完整作品 -> 本地/Qwen 自动筛查 -> 站长审核 -> 上线 -> 下线
                         ^             |                |         |
                         |             v                v         v
                       驳回 <-------- 需复核           驳回       驳回

章节：草稿/定时 -> 本地/Qwen 自动筛查 -> 已发布
                       |                      |
                       v                      v
                    拦截并标记 -------------> 触发整书复核
```

- “定时发布”必须保存计划发布时间，到点后执行自动筛查，不能绕过本地敏感词和 Qwen 审核。
- 驳回应返回可编辑状态和清晰原因。
- 作品连载状态（连载中/已完结）应与书架可见状态分离。
- **D-07 已确认：** 不对每个章节进行独立人工审核。完整作品上线前由站长人工审核；新增或修改章节只经过本地敏感词与 Qwen 自动筛查。命中高风险或审核失败时，暂停章节发布并将作品标记为需站长复核。
- **D-08 已确认：** 机器审核采用本地敏感词规则加 Qwen 模型。后端使用 Spring AI 2.0.x 的 OpenAI 兼容客户端调用百炼 Qwen；模型仅输出可审计的风险建议，不能自动批准上线。

### 5.2 兑换码与权益

```text
兑换码：草稿/导入 -> 可用 -> 已核销
                     |       |
                     v       v
                   已禁用   已发放权益
                     |
                     v
                   已过期
```

- 不接入微信、支付宝或任何线上收款；不存在支付回调、退款、提现或外部对账。
- 兑换码为一次性核销资产，必须具备批次、权益类型、额度/时长、过期时间、状态、核销用户和核销时间。
- 代币、打赏、票、会员和购买权益均需要独立账本记录；账本写入与兑换码状态迁移必须在同一事务中完成。
- **D-09 已确认：** 使用站长生成/导入的兑换码替代线上支付，模式参考卡网和 `new-api`。兑换码可兑换代币、会员有效期或指定作品权益。
- **D-10 已确认：** 首期采用精简商业规则：兑换码一次性使用、可配置过期时间和禁用状态；站长可生成、导入、禁用并审计兑换码；代币用于整本购买和打赏；会员、票和打赏额度均使用后台可配置值。首期不提供退款、提现或财务对账功能。
